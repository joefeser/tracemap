# Site TraceMap Tools Demo Summary Refresh Requirements

Public claim level: hidden

## Objective

Prepare a future site/tooling phase that reduces drift between public demo
counts, public-safe artifact paths, and static site copy. The future work should
give maintainers a bounded way to refresh or validate summary facts derived from
`./scripts/demo-public.sh` output without publishing raw generated internals or
coupling the public site to private local artifacts.

This PR is spec-only. It does not implement the refresh, validation, fixture, or
site-copy changes.

## Why The Claim Level Is Hidden

The refresh mechanism is an internal maintenance tool, not a public feature or
product claim. It may feed demo-level public pages after a maintainer reviews and
commits a scrubbed fixture, but the mechanism itself should not be marketed or
shown as user-facing automation. Public pages that consume refreshed facts must
continue to describe only demo evidence from checked-in samples and public-safe
generated summaries.

## Chosen Approach

The future implementation should use a generated public-safe data fixture with a
small explicit refresh script and a cheap validation check.

This is more useful than validation-only because it gives site pages one shared
source for counts, statuses, coverage labels, and public-safe artifact paths.
It is safer than a build-time script that reads arbitrary local demo output
because the static site build should not depend on private directories, raw scan
artifacts, or a local `./scripts/demo-public.sh` run. The refresh must be an
explicit maintainer action that produces a reviewed, scrubbed fixture committed
under `site/src/`. In the first implementation, affected pages should be
validated against the fixture; adding a site templating/data consumption layer is
out of scope unless a later spec explicitly approves it.

## Requirements

### Requirement 1: Define A Public-Safe Summary Fixture

The future implementation shall introduce a committed fixture that contains only
site-facing fields derived from generated public demo summaries.

Acceptance criteria:

- The fixture is committed under an underscore-prefixed directory in `site/src/`
  so it is not copied into `site/dist/`; suggested path:
  `site/src/_data/demo-public-summary.json`.
- The fixture includes a schema or version field so future changes are explicit.
- The fixture maps from the real `demo-summary.json` section fields: `name`,
  `status`, `classification`, `evidenceTier`, `ruleIds`, `reportCoverage`,
  `artifactPaths`, `counts`, and `reason`.
- The fixture may rename `name` to a documented `id` by using a checked mapping
  table for known section names. Unknown names must fail until the mapping is
  updated.
- The fixture may rename `reportCoverage` to `coverage` and `artifactPaths` to
  `artifacts`, but only for the public-safe subset of `artifactPaths`; the
  mapping must be documented and tested.
- The fixture may include statuses, classification labels, evidence tiers, rule
  IDs, coverage labels, counts, reasons, public-safe report-family paths, and
  generated summary names.
- The fixture must not invent per-section limitations from `demo-summary.json`;
  limitations may be carried only if a bounded extractor reads them from
  approved public-safe report families, or they may remain page prose validated
  against section status and coverage.
- If a source section contains raw local-only artifact paths such as the current
  `sample-scans` row's `scans/.../report.md` paths, the fixture must omit those
  paths from `artifacts` and may record only a generic local-only artifact
  family label without a path.
- The fixture may include a hashed output-root label such as
  `path-hash:<hex>` only if it is already present in `demo-summary.json`.
- The fixture must not include raw snippets, raw SQL, config values, secrets,
  local absolute paths, raw repository remotes, raw `facts.ndjson`,
  `index.sqlite`, combined SQLite files, generated scan directories, analyzer
  logs, copied local reports, or private sample identities.

### Requirement 2: Refresh Only From Approved Public Demo Inputs

The future refresh command shall read only bounded public-safe demo output.

Acceptance criteria:

- The refresh input is an explicit demo output root generated by
  `./scripts/demo-public.sh <ignored-out>`.
- The refresh may read `demo-summary.json` and `demo-summary.md`.
- The refresh may read generated public-safe report Markdown/JSON families only
  when they are selected by the same allowlist shape as `collectPublicFiles()`
  and pass the same leak rejection checks as `findSentinelFailures()`.
- The refresh must not read files under source `artifactPaths` that point to raw
  generated families such as `scans/`; those paths are source metadata only and
  must not be copied into committed fixture artifacts.
- The refresh must not read from `scans/`, `combined/`, `logs/`, raw fact
  streams, SQLite indexes, analyzer logs, generated scan manifests, or copied
  private/local reports.
- The refresh rejects missing `demo-summary.json`, a `version` field that is
  not `1.0` or a later explicitly supported version string, empty section lists,
  missing rule IDs, missing evidence tiers, and missing reasons for deferred,
  unavailable, or failed sections.
- The refresh validates that every `portfolio-manifest.json` `indexPath` value
  is a relative path before treating the file as approved public-safe input.

### Requirement 3: Scrub Or Reject Unsafe Paths And Values

The future refresh and validation logic shall reject unsafe output instead of
silently publishing it.

Acceptance criteria:

- Local absolute paths are rejected, including Unix home paths, temporary
  directories, CI workspace paths, mounted workspace paths, and Windows drive or
  UNC paths.
- Raw repository remotes, `file://` URLs, `.git` paths, private sample names,
  and host-specific output roots are rejected.
- Public artifact paths are normalized to POSIX-style relative paths rooted
  under approved public-safe families such as `demo-summary.md`,
  `demo-summary.json`, `reports/...`, and `portfolio-manifest.json`.
- Any path traversal segment, empty path segment, backslash-only path, or
  generated raw-artifact family in committed fixture data or public page
  references causes the refresh or validation to fail.
- Scrubbing must not replace unsafe values with vague placeholders in committed
  data; unsafe values should fail validation so a maintainer can fix the
  generator or source summary.

### Requirement 4: Avoid Manual Count Drift On Public Pages

The future site implementation shall make affected pages consume the shared
fixture or validate their hard-coded facts against it.

Acceptance criteria:

- `/demo/result/` validates against the fixture for current public demo section
  statuses, counts, evidence tiers, coverage labels, and public-safe artifact
  families.
- `/demo/proof-upgrades/` validates against the fixture for upgraded proof-row
  counts, report coverage, reasons, and generated report paths. Limitation text
  may remain page prose unless limitations are explicitly extracted from
  approved public-safe reports.
- `/demo/proof-assets/` validates against the fixture for public-safe summary,
  proof-asset references, and hard-coded counts shown in page copy.
- `/packets/` validates against the fixture for evidence-packet artifact
  examples and local-only/public-safe boundaries.
- `/manager-packet/` validates against the fixture for manager-facing summary
  counts and proof links.
- `/capabilities/` validates against the fixture for status labels and proof
  paths tied to demo evidence.
- Pages may still contain explanatory prose, but demo counts and generated
  public-safe artifact paths should have one checked source.

### Requirement 5: Preserve Static Site Behavior

The future implementation shall keep `tracemap.tools` a static site.

Acceptance criteria:

- No backend, runtime service, client-side fetch from local files, or dynamic
  public artifact discovery is introduced.
- The site build consumes only checked-in site sources and the committed
  public-safe fixture.
- Local demo output remains outside `site/dist/` and outside the git-tracked
  static output.
- AWS Amplify continues to build from the repository root with `site` as
  `appRoot` and publishes only `site/dist`.
- The refresh command is a maintainer command, not an automatic dependency of
  every normal site build unless the committed fixture already exists.

### Requirement 6: Preserve Public Claim Boundaries

The future implementation shall not upgrade public claims beyond deterministic
static demo evidence.

Acceptance criteria:

- Public copy and generated data do not claim TraceMap proves runtime behavior,
  production traffic, deployment state, endpoint performance, production
  dependency understanding, P1 root cause, release safety, or AI impact
  analysis.
- Public copy keeps rule IDs, evidence tiers, coverage labels, status labels,
  counts, limitations, and generated public-safe artifact names attached to
  any summary fact.
- Deferred, unavailable, failed, reduced, and partial analysis states remain
  visible instead of being hidden by the fixture.

### Requirement 7: Validate Cheaply In Spec-Followup PRs

The future implementation shall include narrow validation for the fixture and
affected site pages.

Acceptance criteria:

- A validation command checks fixture schema, allowed fields, safe path forms,
  safe value forms, and affected page references.
- Existing `site` validation continues to pass.
- `git diff --check` remains part of the validation plan.
- Browser sanity checks are required only when page layout or interaction
  changes, not for fixture-only updates.

## Artifact Safety Rules

Safe future inputs:

- `demo-summary.md`
- `demo-summary.json`
- Public-safe report Markdown/JSON files that pass the existing demo sentinel
  rules
- `portfolio-manifest.json` only when it contains approved relative paths and
  passes the sentinel checks

Safe future committed data:

- Sanitized section IDs and labels
- Status labels, evidence tiers, rule IDs, coverage labels, reasons, counts,
  limitations, and public-safe relative artifact paths
- Summary fixture version and source summary version
- A path hash already generated by the public demo summary

Local-only data:

- Raw source snippets, raw SQL, config values, secrets, local absolute paths,
  raw repository remotes, private sample identities, raw `facts.ndjson`,
  `index.sqlite`, combined SQLite files, generated scan directories, scan
  manifests, analyzer logs, copied local reports, and arbitrary generated
  output trees
