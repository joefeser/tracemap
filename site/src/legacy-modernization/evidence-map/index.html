Legacy modernization evidence map

Modernization planning starts with static evidence, not a safety claim.

This concept page maps legacy modernization questions to deterministic static repository evidence, coverage gaps, hidden surface families, and public-safe proof routes. It works from repository snapshots and checked-in artifacts, not live systems.

Public claim level: concept. No public conclusion without evidence. This page is a reviewer-question map for modernization planning, not the legacy evidence story and not the legacy validation plan.

Coverage gaps

Evidence tier language keeps partial analysis useful.

Tier1SemanticCompiler-resolved static symbol evidence when project load and semantic analysis succeed.
Tier2StructuralKnown framework, project, or file-structure patterns that can guide review with limitations visible.
Tier3SyntaxOrTextualSyntax-only or text evidence used when stronger analysis is unavailable or reduced.
Tier4UnknownAnalysis gaps, unsupported formats, malformed metadata, or unable-to-prove states that block stronger conclusions.

Non-claims

The map is planning context, not modernization approval.

  • No runtime behavior, production traffic, endpoint performance, outage cause, release safety, operational safety, production dependency understanding, exploitability, database existence, package compatibility, migration feasibility, migration completeness, AI impact analysis, LLM analysis, or complete product coverage proof is claimed.
  • Static evidence does not replace runtime telemetry, source-owner review, architecture review, security review, database review, test results, build or restore validation, migration planning, release approval, code review, governance, or human judgment.
  • Rows avoid impact wording unless a deterministic reducer result with public-safe rule IDs, evidence tiers, proof path, and limitations supports that exact statement.
