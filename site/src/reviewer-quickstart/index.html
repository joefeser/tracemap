Reviewer quickstart

Inspect the claim, proof path, and limits before repeating the packet.

This concept guide helps a code reviewer, maintainer, reviewer agent, or engineering owner inspect a TraceMap evidence packet or public-safe proof trail in about five minutes. It is the first-stop orientation, not the complete review workflow.

Public claim level: concept. No public conclusion without evidence. This page is reviewer-facing guidance for static evidence packets, not a release gate, approval system, source-review substitute, test substitute, runtime observability substitute, or automated decision.

Five-Minute Review

Use the same order before a claim is repeated, upgraded, or routed.

1. identify the claim

Name the exact claim first; stop if it bundles several conclusions.

2. find the proof path

Locate a public-safe route, summary, documented source trail, or named private review location; stop if the trail is absent.

3. check public claim level

Keep concept, demo, shipped, hidden, or other supported vocabulary from being upgraded by confidence or repetition.

4. read rule ID/family

Confirm the deterministic rule ID or documented rule family before repeating evidence-backed wording.

5. inspect evidence tier and coverage label

Read Tier1Semantic, Tier2Structural, Tier3SyntaxOrTextual, or Tier4Unknown with the coverage label.

6. check commit/extractor context

Check the commit SHA or source revision and extractor version or extractor family when public-safe; otherwise record the limitation.

7. read limitations/non-claims

Carry limitations and non-claims with the claim instead of treating them as footnotes.

8. name next owner

Route missing validation, source, test, runtime, release, or ownership questions to a human owner category.

9. stop on missing evidence

Do not repeat, upgrade, approve, or route a stronger claim when required evidence is missing.

Stop Conditions

Stop when the packet would make the public claim stronger than the evidence.

  • missing proof path: no public-safe route, summary, documented trail, or named private review location.
  • missing rule ID or rule family: no deterministic rule basis or documented family.
  • missing evidence tier: tier strength is absent or unlabeled.
  • missing coverage label: coverage state is missing from the cited evidence.
  • missing limitation: the claim omits what the evidence cannot prove.
  • missing public claim level: the route or packet does not show concept, demo, shipped, hidden, or another supported level.
  • missing commit or extractor context without explicit limitation: source revision or extractor context is absent and not explained.
  • no validation evidence: no public-safe validation status, test summary, review summary, or command family is named.
  • no next owner: the remaining question has nowhere to go.
  • private-only support presented as public proof: private evidence is treated as public support.
  • raw artifact leakage: raw facts, raw SQLite content, analyzer logs, raw source snippets, raw SQL, config values, secrets, local paths, raw remotes, generated scan directories, private sample names, raw command output, hidden validation details, or credential-like values appear in public copy.
  • unsupported wording: runtime, release, safety, production, AI/LLM, autonomous approval, or complete-coverage language appears outside explicit boundary copy.

Safe Review Language

Use bounded verbs and keep unsupported conclusions out of the sentence.

Useinspect, check, follow, review, compare, label, record, route, escalate, or cannot conclude from this packet.
SayThis packet shows where to inspect the public-safe proof path, rule basis, tier, coverage, context, limitations, and next owner before repeating the claim.
AvoidDo not call a change, endpoint, service, vendor system, team, release, dependency, database reference, or packet approved, blocked, root cause, production-proven, complete, or impacted from this quickstart.
RouteWhen evidence is missing, name the owner question instead of turning absence into confidence.

Non-Claims

The quickstart reads evidence packets; it does not expand what they prove.

  • This quickstart and cited packets do not prove runtime behavior, production traffic, endpoint performance, outage cause, release approval, release safety, operational safety, or complete coverage.
  • They do not perform AI impact analysis, LLM analysis, embeddings, vector database analysis, prompt classification, autonomous approval, or release approval.
  • They do not replace tests, code review, source review, runtime observability, telemetry, service ownership, release controls, or human judgment.
  • The reviewer may inspect, compare, record, route, or escalate; they cannot conclude from this packet when required evidence is missing.
