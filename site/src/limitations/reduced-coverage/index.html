Reduced coverage playbook

When evidence is partial, keep the label attached.

Reduced coverage is an evidence state and a handoff trigger. It tells readers what TraceMap could still inspect, which proof path is incomplete, which owner role should collect the next evidence, and where the public statement must stop.

Public claim level: concept. No public conclusion without evidence. This playbook teaches labeling and owner handoff; it is not a scanner result, reducer result, runtime proof, release gate, or approval flow.

Meaning

Reduced coverage names a boundary, not a conclusion.

Partial evidence can still help reviewRule IDs, evidence tiers, coverage labels, limitations, commit context, extractor-version context, and public-safe proof links can route follow-up when they stay visible.
Fallback evidence stays labeledSyntax fallback, structure matches, config scans, project files, and public-safe gap summaries are review input until semantic or owner evidence is collected.
Missing proof is preservedUnknown, unavailable, private-only, stale, future-only, and analysis-gap labels must not be normalized into stronger wording.

Labeling

Carry the reduced label beside the sentence.

coverage labelUse the visible reduced, partial, syntax fallback, unknown, unavailable, private-only, stale, future-only, or analysis-gap label from the cited surface.
evidence tierUse only Tier1Semantic, Tier2Structural, Tier3SyntaxOrTextual, or Tier4Unknown; private-only or unavailable source evidence uses Tier4Unknown with that marker.
proof contextAttach rule ID or rule family, public-safe source context, extractor-version context, limitation, what cannot be concluded, next owner role, safe wording, and stop condition.

Safe conclusions

Static referenceSyntax fallback found a static reference that needs semantic or owner review.
Missing artifactThe proof path is incomplete because public-safe artifact output is unavailable.
Private-only evidencePrivate-only evidence can support internal follow-up, but public wording waits for a public-safe summary.
Stale contextThe source context is stale, so current-head wording must wait.
Unknown tierThe tier is unknown, so the claim should be downgraded or held.

Unsafe conclusions

Reject patterns that upgrade a gap into a claim.

  • Rejected pattern: reduced coverage proves there is no impact.
  • Rejected pattern: a failed or reduced analysis is a clean repo.
  • Rejected pattern: syntax fallback proves runtime behavior or compiler-resolved impact.
  • Rejected pattern: the release is approved, certified, guaranteed, or safe because no public proof was found.
  • Rejected pattern: operational safety, complete coverage, AI impact analysis, LLM analysis, prompt-based classification, embedding search, vector database analysis, autonomous approval, or replacement of human review is provided here.

Owner handoff

Transfer the evidence question, not approval.

Current labelState the row label, coverage label, and evidence tier exactly as the public-safe surface shows them.
Evidence availableName the public-safe evidence category and the missing evidence without quoting raw artifacts or private detail.
Requested next actionAsk the scanner owner, reviewer, build/tooling owner, framework owner, artifact publisher, site owner, or source owner to collect the next public-safe proof target.
Stop conditionKeep the claim downgraded, held, or internal until the proof target exists and the non-claim still travels with it.
Non-claimThe handoff does not approve a release, prove runtime behavior, assign blame, or replace human review.

Stop conditions

Stop before the wording outruns the evidence.

  • Stop when the proof link is missing, unavailable, private-only, stale, or not public-safe.
  • Stop when the statement drops the coverage label, evidence tier, rule ID or rule family, limitation, or owner follow-up.
  • Stop when a reader asks for runtime behavior, production traffic, endpoint performance, outage cause, release approval, operational safety, or complete coverage.
  • Stop when the next step requires source owner review, build/tooling repair, framework support work, artifact publication, or current source context.

Non-claims

This playbook does not publish raw proof or close the gap.

  • No absence-of-impact proof, clean-repo claim under failed or reduced analysis, runtime proof, production traffic proof, endpoint performance proof, outage-cause proof, release approval, release safety, operational safety, or complete coverage proof.
  • No AI impact analysis, LLM analysis, prompt-based classification, embedding search, vector database analysis, autonomous approval, or replacement of tests, source review, runtime observability, service owner review, or human review.
  • No raw facts, raw SQLite content, analyzer logs, raw source snippets, raw SQL, config values, secrets, local paths, raw remotes, generated scan directories, private sample names, raw command output, hidden validation details, or credential-like values are public material.
  • Artifact families such as fact streams, SQLite indexes, analyzer logs, scan manifests, reports, rule catalog entries, commit metadata, coverage labels, and limitations need public-safe summaries before public linking.
