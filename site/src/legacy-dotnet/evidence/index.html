Legacy .NET evidence lane

Legacy .NET review starts with status rows, not support claims.

This concept lane shows how TraceMap can frame reviewer questions for WCF, ASMX/SOAP, .NET Remoting, WebForms, WinForms, legacy data metadata, project diagnostics, and modernization follow-up using deterministic static evidence language.

Public claim level: concept. No public conclusion without evidence. TraceMap works from repository snapshots and checked-in artifacts, not runtime traffic, production telemetry, live system inspection, or migration execution.

Branch and proof boundary

Status labels describe proof posture, not general availability.

shippedAllowed only when exact wording is true on main and backed by a public-safe proof path.
demoAllowed only for checked-in public-safe demo artifacts, summaries, or validation output that support the exact wording.
devAllowed only when exact wording is true on the target branch and clearly not a main claim.
futureAllowed for reviewer-question or evidence-shape framing without support, detection, or coverage claims.
hiddenUsed when proof, validation detail, or capability scope is not public-safe; rows name only abstract framework families.

Limitations

Static evidence remains bounded even when it is useful.

semantic proofCompiler-resolved Roslyn symbol evidence is stronger than syntax fallback, but still static and repository-bounded.
structural proofKnown framework or project patterns can guide review while keeping framework and toolchain limits visible.
syntax fallbackSyntax-only or textual evidence keeps partial analysis useful, but it cannot resolve compiler symbols.
unknown gapsUnsupported project type, missing toolchain, malformed metadata, or generated-file uncertainty blocks stronger conclusions.

Non-claims

The lane does not publish operational or private proof.

  • No runtime behavior, production traffic, live endpoint existence, endpoint performance, service reachability, UI reachability, user workflow execution, outage cause, release approval, release safety, operational safety, security posture, exploitability, database existence, query execution, schema compatibility, package compatibility, migration feasibility, migration completeness, or complete product coverage is claimed.
  • TraceMap does not replace runtime telemetry, source-owner review, architecture review, security review, database review, test results, build or restore validation, migration planning, release approval, or human judgment.
  • No AI or LLM workflow, embedding workflow, vector database, prompt-based classification, probabilistic ranking, autonomous approval, or agentic release decision is part of this lane.
  • No raw source snippets, raw SQL, config values, connection strings, secrets, tokens, credentials, local paths, raw remotes, generated scan directories, raw facts, raw SQLite indexes, analyzer log content, raw service addresses, raw endpoint values, database identifiers, private sample names, customer names, production identifiers, hidden validation details, hidden capability counts, or unreleased sequencing are published.
  • Artifact names such as scan-manifest.json, facts.ndjson, index.sqlite, report.md, and analyzer logs are output categories here, not raw contents or paths.
