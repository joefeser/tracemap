Demo proof assets

Visuals can make proof scannable without becoming the proof.

This page shows public-safe, screenshot-style representations of the public demo report outputs: summary rows, static paths, before/after change rows, impact rows, and release-review checklist items.

Public claim level: demo. These visuals are orientation aids over deterministic static evidence. Generated reports, rule IDs, evidence tiers, coverage labels, limitations, facts, and indexes remain the source of truth.

Artifact safety

What this page can show, and what must stay local.

  • Safe: public demo row names, statuses, rule IDs, evidence tiers, coverage labels, demo counts, report-family names, and limitation labels.
  • Safe: visual representations of public-safe report summaries when they remain tied to evidence and limitations.
  • Local-only: raw source snippets, raw SQL, config values, secrets, local absolute paths, raw repository remotes, and private sample identities.
  • Local-only: raw fact streams, SQLite indexes, combined SQLite files, generated scan directories, analyzer logs, and copied local report archives.
  • Not safe: claiming the visual board proves full impact, runtime reachability, production dependency understanding, endpoint performance, or release approval.

The visual board is a map legend, not the territory.

Use it to help reviewers recognize the shape of a demo evidence packet. Then follow the generated reports, rule IDs, evidence tiers, coverage labels, counts, and limitations before making any public conclusion.

For incident-adjacent readers, use visuals only to orient.

The incident review orientation concept explains how these public-safe demo visuals can help a team recognize static evidence shapes before inspecting generated reports. The visuals do not prove runtime cause, production behavior, endpoint performance, or release safety.
